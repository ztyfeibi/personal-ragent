# Ragent 项目架构分析文档计划

## Summary

产出一份面向源码的架构分析文档，采用“两阶段组织”：
先分别分析各个模块，再把模块分析汇总为整体架构分析。
文档目标不是做实现细节解释，而是建立一套能从源码快速定位职责、边界、依赖关系和调用路径的阅读索引。

默认分析对象包括：
`bootstrap`、`framework`、`infra-ai`、`mcp-server`、`frontend`，并在总体分析中补充 `scripts`、`assets/docs`、根级 `pom.xml` 与配置资源对架构的支撑作用。

## Key Changes

### 1. 文档整体结构

文档按以下主结构编排：

1. 项目总览
2. 模块级分析
3. 模块协同分析
4. 关键调用流程
5. 附录：阅读顺序与源码导航建议

项目总览先说明：

- 技术栈与分层
- Maven 多模块边界
- 后端、前端、基础设施、工具脚本的职责定位
- “源码文件级分析”采用何种粒度呈现

### 2. 模块级分析模板

每个模块统一按同一模板展开，保证后续能横向对比：

- 模块作用：回答“这个模块解决什么问题”
- 包/目录结构：按源码目录列出一级或二级包职责
- 文件级说明：覆盖该模块下所有源码文件，按包分组逐个说明用途
- 对外提供的能力：控制器、服务、SPI、工具、配置、实体、枚举、切面、前端页面/组件等
- 依赖与被依赖关系：说明它依赖谁、被谁调用、通过什么接口耦合
- 模块内部主链路：从入口到核心处理再到输出的局部调用流程

模块重点建议如下：

- `bootstrap`：按 `admin/core/ingestion/knowledge/rag/user` 六个业务域拆解
- `framework`：按横切能力拆解，如异常、上下文、trace、幂等、SSE、MQ、缓存、数据库
- `infra-ai`：按模型接入能力拆解，如 `chat`、`embedding`、`rerank`、`token`、`http`、`config`
- `mcp-server`：按 MCP 服务暴露能力、工具注册与桥接方式拆解
- `frontend`：按页面、组件、hooks、services、stores、types 的职责拆解

### 3. 总体分析部分

在模块分析之后，增加一层“跨模块视角”的总体分析，重点回答：

- 模块之间如何耦合协作
- 哪些是稳定边界，哪些是高频交互边界
- 依赖方向是否清晰，是否存在典型的分层穿透
- 前后端、业务层、基础设施层、AI 能力层如何串起来
- 配置、数据库、Redis、Milvus、RocketMQ、S3 在整体架构中的位置

这一部分不再逐文件罗列，而是按协作关系组织：

- `frontend -> bootstrap`
- `bootstrap -> framework`
- `bootstrap -> infra-ai`
- `bootstrap -> 存储/向量库/消息队列/缓存`
- `mcp-server` 在整套系统中的独立性与接入点

### 4. 整体调用流程部分

最后抽出 4 条全局主流程，作为整套架构的“动态视图”：

- 用户发起 SSE 问答流程
- RAG 检索与生成流程
- 文档上传、切块、向量化、入库流程
- 管理后台配置/知识库管理驱动后端变更流程

每条流程统一说明：

- 入口模块与入口文件
- 中间关键服务/组件
- 模块切换点
- 外部基础设施参与点
- 输出结果与状态落点

## Deliverable Shape

最终文档建议写成单一 Markdown 文件，默认命名：
`docs/ragent-architecture-analysis.md`

文档呈现原则：

- 模块分析优先，整体分析次之
- 文件级说明按包分组，不做纯目录清单
- 每个文件只写“职责 + 在链路中的位置”，避免复制源码实现
- 总体流程用编号步骤描述，必要时补 1 张总览关系图占位说明

## Test Plan

文档完成后按以下标准自检：

- 能覆盖所有主要模块，不遗漏 `mcp-server` 与 `frontend`
- 每个模块都回答四个问题：作用、文件用途、模块耦合、局部流程
- 总体分析不是模块摘要拼接，而是有明确跨模块协同说明
- 至少包含 4 条全局调用流程
- 读者不看源码，也能先建立架构心智模型；看源码时，又能据此快速定位文件

## Assumptions

- 当前阶段只输出“架构分析计划”，不直接写正文文档
- “所有源码文件”按模块内源码文件覆盖，不包含构建产物、依赖缓存、图片资源本体
- 文件级说明以“源码阅读导引”为目标，不展开到逐方法说明
- 文档以中文编写，面向熟悉 Java/Spring Boot/React 的开发者
